17    /**
18    * <p>Provides support classes for multi-threaded programming.
19    * This package is intended to be an extension to {@link java.util.concurrent}.
20    * These classes are thread-safe.</p>
21    *
22    * <h3></h3>
23    *
24    * <p> A group of classes deals with the correct creation and initialization of objects that are accessed by multiple threads.
25    * All these classes implement the {@link org.apache.commons.lang3.concurrent.ConcurrentInitializer} interface which provides just a
26    * single method:
27    * </p>
28    *
29    * <pre>
30    * <code>
31    * public interface ConcurrentInitializer&lt;T&gt; {
32    * T get() throws ConcurrentException;
33    * }
34    * </code>
35    * </pre>
36    *
37    * <p>A <code>ConcurrentInitializer</code> produces an object.
38    * By calling the {@link org.apache.commons.lang3.concurrent.ConcurrentInitializer#get() get()} method the object managed by the initializer can be obtained.
39    * There are different implementations of the interface available
40    * addressing various use cases:
41    * </p>
42    *
43    * <p> {@link org.apache.commons.lang3.concurrent.ConstantInitializer} is a very straightforward implementation of the <code>ConcurrentInitializer</code> interface:
44    * An instance is passed an object when it is constructed.
45    * In its <code>get()</code> method it simply returns this object.
46    * This is useful, for instance in unit tests or in cases when you want to pass a specific object to a component which expects a <code>ConcurrentInitializer</code>.
47    * </p>
48    *
49    * <p>The {@link org.apache.commons.lang3.concurrent.LazyInitializer} class can be used to defer the creation of an object until it is actually used.
50    * This makes sense, for instance, if the creation of the object is expensive and would slow down application startup or if the object is needed only for special executions.
51    * <code>LazyInitializer</code> implements the <em>double-check idiom for an instance field</em> as discussed in Joshua Bloch's "Effective Java", 2nd edition, item 71.
52    * It uses <strong>volatile</strong> fields to reduce the amount of synchronization.
53    * Note that this idiom is appropriate for instance fields only.
54    * For <strong>static</strong> fields there are superior alternatives.</p>
55    *
56    * <p>We provide an example use case to demonstrate the usage of this class:
57    * A server application uses multiple worker threads to process client requests.
58    * If such a request causes a fatal error, an administrator is to be notified using a special messaging service.
59    * We assume that the creation of the messaging service is an expensive operation.
60    * So it should only be performed if an error actually occurs.
61    * Here is where <code>LazyInitializer</code> comes into play.
62    * We create a specialized subclass for creating and initializing an instance of our messaging service.
63    * <code>LazyInitializer</code> declares an abstract {@link org.apache.commons.lang3.concurrent.LazyInitializer#initialize() initialize()} method which we have to implement to create the messaging service object:</p>
64    *
65    * <pre>
66    * <code>
67    * public class MessagingServiceInitializer extends LazyInitializer&lt;MessagingService&gt; {
68    * protected MessagingService initialize() throws ConcurrentException {
69    * // Do all necessary steps to create and initialize the service object
70    * MessagingService service = ...
71    * return service;
72    * }
73    * }
74    * </code>
75    * </pre>
76    *
77    * <p> Now each server thread is passed a reference to a shared instance of our new <code>MessagingServiceInitializer</code> class.
78    * The threads run in a loop processing client requests. If an error is detected, the messaging service is obtained from the initializer, and the administrator is notified:</p>
79    *
80    * <pre>
81    * <code>
82    * public class ServerThread implements Runnable {
83    * // The initializer for obtaining the messaging service.
84    * private final ConcurrentInitializer&lt;MessagingService&gt; initializer;
85    *
86    * public ServerThread(ConcurrentInitializer&lt;MessagingService&gt; init) {
87    * initializer = init;
88    * }
89    *
90    * public void run() {
91    * while (true) {
92    * try {
93    * // wait for request
94    * // process request
95    * } catch (FatalServerException ex) {
96    * // get messaging service
97    * try {
98    * MessagingService svc = initializer.get();
99    * svc.notifyAdministrator(ex);
100    * } catch (ConcurrentException cex) {
101    * cex.printStackTrace();
102    * }
103    * }
104    * }
105    * }
106    * }
107    * </code>
108    * </pre>
109    *
110    * <p>The {@link org.apache.commons.lang3.concurrent.AtomicInitializer} class is very similar to <code>LazyInitializer</code>.
111    * It serves the same purpose: to defer the creation of an object until it is needed.
112    * The internal structure is also very similar.
113    * Again there is an abstract {@link org.apache.commons.lang3.concurrent.AtomicInitializer#initialize() initialize()} method which has to be implemented by concrete subclasses in order to create and initialize the managed object.
114    * Actually, in our example above we can turn the <code>MessagingServiceInitializer</code> into an atomic initializer by simply changing the <strong>extends</strong> declaration to refer to <code>AtomicInitializer&lt;MessagingService&gt;</code> as super class.</p>
115    *
116    * <p>With {@link org.apache.commons.lang3.concurrent.AtomicSafeInitializer} there is yet another variant implementing the lazy initializing pattern.
117    * Its implementation is close to <code>AtomicInitializer</code>; it also uses atomic variables internally and therefore does not need synchronization.
118    * The name &quot;Safe&quot; is derived from the fact that it implements an additional check which guarantees that the {@link org.apache.commons.lang3.concurrent.AtomicSafeInitializer#initialize() initialize()} method is called only once.
119    * So it behaves exactly in the same way as <code>LazyInitializer</code>.</p>
120    *
121    * <p>Now, which one of the lazy initializer implementations should you use?
122    * First of all we have to state that is is problematic to give general recommendations regarding the performance of these classes.
123    * The initializers make use of low-level functionality whose efficiency depends on multiple factors including the target platform and the number of concurrent threads.
124    * So developers should make their own benchmarks in scenarios close to their specific use cases.
125    * The following statements are rules of thumb which have to be verified in practice.</p>
126    *
127    * <p><code>AtomicInitializer</code> is probably the most efficient implementation due to its lack of synchronization and further checks.
128    * Its main drawback is that the <code>initialize()</code> method can be called multiple times.
129    * In cases where this is not an issue <code>AtomicInitializer</code> is a good choice.
130    * <code>AtomicSafeInitializer</code> and <code>LazyInitializer</code> both guarantee that the initialization method is called only once.
131    * Because <code>AtomicSafeInitializer</code> does not use synchronization it is probably slightly more efficient than <code>LazyInitializer</code>, but the concrete numbers might depend on the level of concurrency.</p>
132    *
133    * <p>Another implementation of the <code>ConcurrentInitializer</code> interface is {@link org.apache.commons.lang3.concurrent.BackgroundInitializer}.
134    * It is again an abstract base class with an {@link org.apache.commons.lang3.concurrent.BackgroundInitializer#initialize() initialize()} method that has to be defined by concrete subclasses.
135    * The idea of <code>BackgroundInitializer</code> is that it calls the <code>initialize()</code> method in a separate worker thread.
136    * An application creates a background initializer and starts it.
137    * Then it can continue with its work while the initializer runs in parallel.
138    * When the application needs the results of the initializer it calls its <code>get()</code> method.
139    * <code>get()</code> blocks until the initialization is complete.
140    * This is useful for instance at application startup.
141    * Here initialization steps (e.g. reading configuration files, opening a database connection, etc.) can be run in background threads while the application shows a splash screen and constructs its UI.</p>
142    *
143    * <p>As a concrete example consider an application that has to read the content of a URL - maybe a page with news - which is to be displayed to the user after login.
144    * Because loading the data over the network can take some time a specialized implementation of <code>BackgroundInitializer</code> can be created for this purpose:</p>
145    *
146    * <pre>
147    * <code>
148    * public class URLLoader extends BackgroundInitializer&lt;String&gt; {
149    * // The URL to be loaded.
150    * private final URL url;
151    *
152    * public URLLoader(URL u) {
153    * url = u;
154    * }
155    *
156    * protected String initialize() throws ConcurrentException {
157    * try {
158    * InputStream in = url.openStream();
159    * // read content into string
160    * ...
161    * return content;
162    * } catch (IOException ioex) {
163    * throw new ConcurrentException(ioex);
164    * }
165    * }
166    * }
167    * </code>
168    * </pre>
169    *
170    * <p>An application creates an instance of <code>URLLoader</code> and starts it.
171    * Then it can do other things.
172    * When it needs the content of the URL it calls the initializer's <code>get()</code> method:</p>
173    *
174    * <pre>
175    * <code>
176    * URL url = new URL("http://www.application-home-page.com/");
177    * URLLoader loader = new URLLoader(url);
178    * loader.start(); // this starts the background initialization
179    *
180    * // do other stuff
181    * ...
182    * // now obtain the content of the URL
183    * String content;
184    * try {
185    * content = loader.get(); // this may block
186    * } catch (ConcurrentException cex) {
187    * content = "Error when loading URL " + url;
188    * }
189    * // display content
190    * </code>
191    * </pre>
192    *
193    * <p> Related to <code>BackgroundInitializer</code> is the {@link org.apache.commons.lang3.concurrent.MultiBackgroundInitializer} class.
194    * As the name implies, this class can handle multiple initializations in parallel.
195    * The basic usage scenario is that a <code>MultiBackgroundInitializer</code> instance is created.
196    * Then an arbitrary number of <code>BackgroundInitializer</code> objects is added using the {@link org.apache.commons.lang3.concurrent.MultiBackgroundInitializer#addInitializer(String, BackgroundInitializer)} method.
197    * When adding an initializer a string has to be provided which is later used to obtain the result for this initializer.
198    * When all initializers have been added the {@link org.apache.commons.lang3.concurrent.MultiBackgroundInitializer#start()} method is called.
199    * This starts processing of all initializers.
200    * Later the <code>get()</code> method can be called.
201    * It waits until all initializers have finished their initialization.
202    * <code>get()</code> returns an object of type {@link org.apache.commons.lang3.concurrent.MultiBackgroundInitializer.MultiBackgroundInitializerResults}.
203    * This object provides information about all initializations that have been performed.
204    * It can be checked whether a specific initializer was successful or threw an exception.
205    * Of course, all initialization results can be queried.</p>
206    *
207    * <p>With <code>MultiBackgroundInitializer</code> we can extend our example to perform multiple initialization steps.
208    * Suppose that in addition to loading a web site we also want to create a JPA entity manager factory and read a configuration file.
209    * We assume that corresponding <code>BackgroundInitializer</code> implementations exist.
210    * The following example fragment shows the usage of <code>MultiBackgroundInitializer</code> for this purpose:</p>
211    *
212    * <pre>
213    * <code>
214    * MultiBackgroundInitializer initializer = new MultiBackgroundInitializer();
215    * initializer.addInitializer("url", new URLLoader(url));
216    * initializer.addInitializer("jpa", new JPAEMFInitializer());
217    * initializer.addInitializer("config", new ConfigurationInitializer());
218    * initializer.start(); // start background processing
219    *
220    * // do other interesting things in parallel
221    * ...
222    * // evaluate the results of background initialization
223    * MultiBackgroundInitializer.MultiBackgroundInitializerResults results =
224    * initializer.get();
225    * String urlContent = (String) results.getResultObject("url");
226    * EntityManagerFactory emf =
227    * (EntityManagerFactory) results.getResultObject("jpa");
228    * ...
229    * </code>
230    * </pre>
231    *
232    * <p>The child initializers are added to the multi initializer and are assigned a unique name.
233    * The object returned by the <code>get()</code> method is then queried for the single results using these unique names.</p>
234    *
235    * <p> If background initializers - including <code>MultiBackgroundInitializer</code> - are created using the standard constructor, they create their own {@link java.util.concurrent.ExecutorService} which is used behind the scenes to execute the worker tasks.
236    * It is also possible to pass in an <code>ExecutorService</code> when the initializer is constructed.
237    * That way client code can configure the <code>ExecutorService</code> according to its specific needs; for instance, the number of threads available could be limited.</p>
238    *
239    * <h3>Utility Classes</h3>
240    *
241    * <p>Another group of classes in the new <code>concurrent</code> package offers some generic functionality related to concurrency.
242    * There is the {@link org.apache.commons.lang3.concurrent.ConcurrentUtils} class with a bunch of static utility methods.
243    * One focus of this class is dealing with exceptions thrown by JDK classes.
244    * Many JDK classes of the executor framework throw exceptions of type {@link java.util.concurrent.ExecutionException} if something goes wrong.
245    * The root cause of these exceptions can also be a runtime exception or even an error.
246    * In typical Java programming you often do not want to deal with runtime exceptions directly; rather you let them fall through the hierarchy of method invocations until they reach a central exception handler.
247    * Checked exceptions in contrast are usually handled close to their occurrence.
248    * With <code>ExecutionException</code> this principle is violated.
249    * Because it is a checked exception, an application is forced to handle it even if the cause is a runtime exception.
250    * So you typically have to inspect the cause of the <code>ExecutionException</code> and test whether it is a checked exception which has to be handled. If this is not the case, the causing exception can be rethrown.
251    * </p>
252    *
253    * <p>The {@link org.apache.commons.lang3.concurrent.ConcurrentUtils#extractCause(java.util.concurrent.ExecutionException)} method does this work for you.
254    * It is passed an <code>ExecutionException</code> and tests its root cause.
255    * If this is an error or a runtime exception, it is directly rethrown.
256    * Otherwise, an instance of {@link org.apache.commons.lang3.concurrent.ConcurrentException} is created and initialized with the root cause
257    * (<code>ConcurrentException</code> is a new exception class in the <code>o.a.c.l.concurrent</code> package).
258    * So if you get such a <code>ConcurrentException</code>, you can be sure that the original cause for the <code>ExecutionException</code> was a checked exception.
259    * For users who prefer runtime exceptions in general there is also an {@link org.apache.commons.lang3.concurrent.ConcurrentUtils#extractCauseUnchecked(java.util.concurrent.ExecutionException)} method which behaves like <code>extractCause()</code>, but returns the unchecked exception {@link org.apache.commons.lang3.concurrent.ConcurrentRuntimeException} instead.</p>
260    *
261    * <p>In addition to the <code>extractCause()</code> methods there are corresponding {@link org.apache.commons.lang3.concurrent.ConcurrentUtils#handleCause(java.util.concurrent.ExecutionException)} and {@link org.apache.commons.lang3.concurrent.ConcurrentUtils#handleCauseUnchecked(java.util.concurrent.ExecutionException)} methods.
262    * These methods extract the cause of the passed in <code>ExecutionException</code> and throw the resulting <code>ConcurrentException</code> or <code>ConcurrentRuntimeException</code>.
263    * This makes it easy to transform an <code>ExecutionException</code> into a <code>ConcurrentException</code> ignoring unchecked exceptions:</p>
264    *
265    * <pre>
266    * <code>
267    * Future&lt;Object&gt; future = ...;
268    * try {
269    * Object result = future.get();
270    * ...
271    * } catch (ExecutionException eex) {
272    * ConcurrentUtils.handleCause(eex);
273    * }
274    * </code>
275    * </pre>
276    *
277    * <p>There is also some support for the concurrent initializers introduced in the last sub section.
278    * The <code>initialize()</code> method is passed a <code>ConcurrentInitializer</code> object and returns the object created by this initializer.
279    * It is null-safe.
280    * The <code>initializeUnchecked()</code> method works analogously, but a <code>ConcurrentException</code> throws by the initializer is rethrown as a <code>ConcurrentRuntimeException</code>.
281    * This is especially useful if the specific <code>ConcurrentInitializer</code> does not throw checked exceptions.
282    * Using this method the code for requesting the object of an initializer becomes less verbose.
283    * The direct invocation looks as follows:</p>
284    *
285    * <pre>
286    * <code>
287    * ConcurrentInitializer&lt;MyClass&gt; initializer = ...;
288    * try {
289    * MyClass obj = initializer.get();
290    * // do something with obj
291    * } catch (ConcurrentException cex) {
292    * // exception handling
293    * }
294    * </code>
295    * </pre>
296    *
297    * <p>Using the {@link org.apache.commons.lang3.concurrent.ConcurrentUtils#initializeUnchecked(ConcurrentInitializer)} method, this becomes:</p>
298    *
299    * <pre>
300    * <code>
301    * ConcurrentInitializer&lt;MyClass&gt; initializer = ...;
302    * MyClass obj = ConcurrentUtils.initializeUnchecked(initializer);
303    * // do something with obj
304    * </code>
305    * </pre>
306    *
307    * <p>Another utility class deals with the creation of threads.
308    * When using the <em>Executor</em> framework new in JDK 1.5 the developer usually does not have to care about creating threads; the executors create the threads they need on demand.
309    * However, sometimes it is desired to set some properties of the newly created worker threads.
310    * This is possible through the {@link java.util.concurrent.ThreadFactory} interface; an implementation of this interface has to be created and passed to an executor on creation time.
311    * Currently, the JDK does not provide an implementation of <code>ThreadFactory</code>, so one has to start from scratch. </p>
312    *
313    * <p> With {@link org.apache.commons.lang3.concurrent.BasicThreadFactory} Commons Lang has an implementation of <code>ThreadFactory</code> that works out of the box for many common use cases.
314    * For instance, it is possible to set a naming pattern for the new threads, set the daemon flag and a priority, or install a handler for uncaught exceptions.
315    * Instances of <code>BasicThreadFactory</code> are created and configured using the nested {@link org.apache.commons.lang3.concurrent.BasicThreadFactory.Builder} class.
316    * The following example shows a typical usage scenario:</p>
317    *
318    * <pre>
319    * <code>
320    * BasicThreadFactory factory = new BasicThreadFactory.Builder()
321    * .namingPattern("worker-thread-%d")
322    * .daemon(true)
323    * .uncaughtExceptionHandler(myHandler)
324    * .build();
325    * ExecutorService exec = Executors.newSingleThreadExecutor(factory);
326    * </code>
327    * </pre>
328    *
329    * <p>The nested <code>Builder</code> class defines some methods for configuring the new <code>BasicThreadFactory</code> instance.
330    * Objects of this class are immutable, so these attributes cannot be changed later.
331    * The naming pattern is a string which can be passed to <code>String.format()</code>.
332    * The placeholder <em>%d</em> is replaced by an increasing counter value.
333    * An instance can wrap another <code>ThreadFactory</code> implementation; this is achieved by calling the builder's {@link org.apache.commons.lang3.concurrent.BasicThreadFactory.Builder#wrappedFactory(java.util.concurrent.ThreadFactory) wrappedFactory(ThreadFactory)} method.
334    * This factory is then used for creating new threads; after that the specific attributes are applied to the new thread.
335    * If no wrapped factory is set, the default factory provided by the JDK is used.</p>
336    *
337    * <h3>Synchronization objects</h3>
338    *
339    * <p>The <code>concurrent</code> package also provides some support for specific synchronization problems with threads.</p>
340    *
341    * <p> {@link org.apache.commons.lang3.concurrent.TimedSemaphore} allows restricted access to a resource in a given time frame.
342    * Similar to a semaphore, a number of permits can be acquired.
343    * What is new is the fact that the permits available are related to a given time unit.
344    * For instance, the timed semaphore can be configured to allow 10 permits in a second.
345    * Now multiple threads access the semaphore and call its {@link org.apache.commons.lang3.concurrent.TimedSemaphore#acquire()} method.
346    * The semaphore keeps track about the number of granted permits in the current time frame.
347    * Only 10 calls are allowed; if there are further callers, they are blocked until the time frame (one second in this example) is over.
348    * Then all blocking threads are released, and the counter of available permits is reset to 0.
349    * So the game can start anew.</p>
350    *
351    * <p> What are use cases for <code>TimedSemaphore</code>?
352    * One example is to artificially limit the load produced by multiple threads.
353    * Consider a batch application accessing a database to extract statistical data.
354    * The application runs multiple threads which issue database queries in parallel and perform some calculation on the results.
355    * If the database to be processed is huge and is also used by a production system, multiple factors have to be balanced:
356    * On one hand, the time required for the statistical evaluation should not take too long.
357    * Therefore you will probably use a larger number of threads because most of its life time a thread will just wait for the database to return query results.
358    * On the other hand, the load on the database generated by all these threads should be limited so that the responsiveness of the production system is not affected.
359    * With a <code>TimedSemaphore</code> object this can be achieved.
360    * The semaphore can be configured to allow e.g. 100 queries per second.
361    * After these queries have been sent to the database the threads have to wait until the second is over - then they can query again.
362    * By fine-tuning the limit enforced by the semaphore a good balance between performance and database load can be established.
363    * It is even possible to chang? the number of available permits at runtime.
364    * So this number can be reduced during the typical working hours and increased at night.</p>
365    *
366    * <p>The following code examples demonstrate parts of the implementation of such a scenario.
367    * First the batch application has to create an instance of <code>TimedSemaphore</code> and to initialize its properties with default values:</p>
368    *
369    * <code>TimedSemaphore semaphore = new TimedSemaphore(1, TimeUnit.SECONDS, 100);</code>
370    *
371    * <p>Here we specify that the semaphore should allow 100 permits in one second.
372    * This is effectively the limit of database queries per second in our example use case.
373    * Next the server threads issuing database queries and performing statistical operations can be initialized.
374    * They are passed a reference to the semaphore at creation time. Before they execute a query they have to acquire a permit.</p>
375    *
376    * <pre>
377    * <code>
378    * public class StatisticsTask implements Runnable {
379    * // The semaphore for limiting database load.
380    * private final TimedSemaphore semaphore;
381    *
382    * public StatisticsTask(TimedSemaphore sem, Connection con) {
383    * semaphore = sem;
384    * ...
385    * }
386    *
387    * //The main processing method. Executes queries and evaluates their results.
388    * public void run() {
389    * try {
390    * while (!isDone()) {
391    * semaphore.acquire(); // enforce the load limit
392    * executeAndEvaluateQuery();
393    * }
394    * } catch (InterruptedException iex) {
395    * // fall through
396    * }
397    * }
398    * }
399    * </code>
400    * </pre>
401    *
402    * <p>The important line here is the call to <code>semaphore.acquire()</code>.
403    * If the number of permits in the current time frame has not yet been reached, the call returns immediately.
404    * Otherwise, it blocks until the end of the time frame.
405    * The last piece missing is a scheduler service which adapts the number of permits allowed by the semaphore according to the time of day.
406    * We assume that this service is pretty simple and knows only two different time slots:
407    * working shift and night shift.
408    * The service is triggered periodically.
409    * It then determines the current time slot and configures the timed semaphore accordingly.</p>
410    *
411    * <pre>
412    * <code>
413    * public class SchedulerService {
414    * // The semaphore for limiting database load.
415    * private final TimedSemaphore semaphore;
416    * ...
417    *
418    * // Configures the timed semaphore based on the current time of day. This method is called periodically.
419    * public void configureTimedSemaphore() {
420    * int limit;
421    * if (isWorkshift()) {
422    * limit = 50; // low database load
423    * } else {
424    * limit = 250; // high database load
425    * }
426    *
427    * semaphore.setLimit(limit);
428    * }
429    * }
430    * </code>
431    * </pre>
432    *
433    * <p>With the {@link org.apache.commons.lang3.concurrent.TimedSemaphore#setLimit(int)} method the number of permits allowed for a time frame can be changed.
434    * There are some other methods for querying the internal state of a timed semaphore.
435    * Also some statistical data is available, e.g. the average number of <code>acquire()</code> calls per time frame.
436    * When a timed semaphore is no more needed, its <code>shutdown()</code> method has to be called.</p>
437    */
