27    /**
28    * Associates an {@link OWLOntologyID} with ontology-less change data.<br>
29    * An {@link OWLOntologyChangeRecord} captures information about an
30    * {@link OWLOntologyChange} in a way that does not require a reference to an
31    * {@link OWLOntology} object. It does this by referencing an
32    * {@link OWLOntologyID} instead of referencing an {@link OWLOntology}. The
33    * primary reason for doing this is so that changes can be serialized and logged
34    * more easily. It should be kept in mind that {@link OWLOntologyChangeRecord}
35    * objects can represent changes for which there might be no in memory
36    * representation of a specific {@link OWLOntology}. This is also true if an
37    * {@link OWLOntology} object has its {@link OWLOntologyID} changed.<br>
38    * An {@link OWLOntologyChange} object contains two important pieces of
39    * information:
40    * <ol>
41    * <li>The {@link OWLOntologyID} that identifies the ontology that the change
42    * pertains to.</li>
43    * <li>The {@link OWLOntologyChangeData} that describes the change specific
44    * data. For each kind of {@link OWLOntologyChange} there is a corresponding
45    * {@link OWLOntologyChangeData} class which captures the essential details that
46    * pertain to the change. The reason for this separation is that it allows
47    * change information to be captured where the context of the change (the
48    * ontology) is known via some other mechanism.</li>
49    * </ol>
50    * {@code OWLOntologyChangeRecord} objects are immutable.
51    *
52    * @author Matthew Horridge, Stanford University, Bio-Medical Informatics
53    * Research Group
54    * @since 3.3
55    */
